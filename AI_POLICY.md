# Automation / AI Usage Policy

> [!IMPORTANT]
> **Human review needed.** This file is AI-generated prose and has not yet been
> confirmed by a human. Once a human has reviewed and confirmed it, remove this
> notice (see Principle 1).

> [!IMPORTANT]
> **This repository was generated with AI assistance.** It must be reviewed by a
> human and improved upon manually by a human before being relied upon in
> production. Treat everything here as a starting point, not a finished product.

This project welcomes AI-assisted contributions. The principles below are
adapted from the Nixpkgs Automation/AI policy, **with deliberate modifications**
(see "Deviation from Nixpkgs" below): prose may be AI-generated, but it must be
marked for human review, and the code itself must be human work.

## Principles

1. **(Top rule) AI-generated prose must be marked for human review.** Any
   non-code content produced by AI — code comments, doc comments, Markdown, and
   other documentation — must carry a review marker at the top of the file (or
   immediately above the affected block). For Markdown:

   ```markdown
   > [!IMPORTANT]
   > **Human review needed.** AI-generated; not yet confirmed by a human.
   ```

   For source files, use the native comment form so the marker is inert:

   ```ts
   // [!IMPORTANT] Human review needed — AI-generated, unreviewed. See AI_POLICY.md.
   ```

   A human must then review the content and confirm it is accurate. **Once
   confirmed, the marker is removed.** The presence of this marker means the
   prose is provisional; its absence means a human has vouched for it. Run
   `just review` (or `bash scripts/review-markers.sh`) to list files still
   pending.

2. **A responsible human must be in the loop.** Every contribution — code,
   documentation, and communication — must have a person who takes
   responsibility for it. You remain accountable for what you submit regardless
   of which tools produced it.

3. **Code must be human work.** The substance of the code — its logic, design,
   and correctness — must be written and genuinely understood by a human.
   Trivial AI assistance is fine: line-level autocomplete, filling in obvious
   syntax, boilerplate, and mechanical refactors. What is **not** permitted is
   "vibe coding": committing AI-generated logic you have not written or do not
   fully understand.

4. **Prose may be AI-generated.** Code comments, doc comments, Markdown files,
   and other documentation **may** be generated by AI, provided (a) it is marked
   per Principle 1 until reviewed, and (b) a human confirms it correctly
   describes the human-authored code. This is where this policy intentionally
   diverges from Nixpkgs.

5. **Disclose non-trivial automation.** Any non-trivial use of automation must
   be disclosed transparently.
   - For AI-assisted commits, add an `Assisted-by:` git trailer naming the tool
     and model, e.g.:

     ```
     Assisted-by: Claude Code (claude-opus-4-8)
     ```

   - Pull request summaries and review comments are disclosed separately from
     commits.

6. **Automation is never the sole arbiter of a merge.** A change must not be
   merged, or a review resolved, purely on the output of an automated tool. A
   human makes the final call.

## Deviation from Nixpkgs

The Nixpkgs policy requires that human communication — including documentation
and code comments — **remain human-authored**. This project **relaxes that
specific requirement**: prose and comments may be AI-generated (Principle 4),
**but** every such piece must be flagged for human review until confirmed
(Principle 1). The stricter bar is instead placed on the **code itself**
(Principle 3), which must be human work. Everything else follows Nixpkgs.

## Exemptions

Standard editor tooling (completion, formatting, linting), routine maintenance
bots, AI-assisted research or testing that does not directly produce the
submitted artifact, and machine translation are exempt from the disclosure
requirement.
